# VoltChat v1.0 ‚Äî Development Log

> Complete record of how v1 was built, decisions made, and architecture chosen. Reference for future development.

---

## Overview

VoltChat is an anonymous, ephemeral real-time chat application. It was built in 8 phases using Next.js 16, Tailwind CSS v4, and Supabase Realtime. The core principle is **zero data persistence** ‚Äî messages exist only in connected clients' memory and are relayed through Supabase's WebSocket infrastructure.

Originally named "GhostChat", renamed to "VoltChat" during development.

---

## Architecture Decisions

### Why Supabase with no database?
Supabase Realtime provides two features that require no database tables:
- **Broadcast**: Relays arbitrary JSON payloads between connected clients via WebSocket channels. Messages are never stored server-side.
- **Presence**: CRDT-based tracking of connected users per channel. Shows who's online in real-time.

This means the entire app runs on Supabase's free tier with zero SQL setup.

### Why Tailwind v4 CSS-first?
The project was scaffolded with Next.js 16 which ships Tailwind v4. Unlike v3, theme customization uses `@theme inline` blocks in `globals.css` instead of `tailwind.config.ts`. All custom colors, animations, keyframes, and shadows are defined in CSS.

### Why no authentication?
By design ‚Äî VoltChat is anonymous. Users pick a temporary username when joining a room. No accounts, no sessions, no cookies.

### Public rooms via Presence (no database)
Public rooms are discovered through a global "lobby" Presence channel. When a creator makes a public room, they track their room metadata in the lobby. When they leave, the room automatically disappears. This keeps the zero-persistence promise.

## Phase-by-Phase Development

### Phase 1 ‚Äî Foundation & Setup
- Scaffolded Next.js 16 project with TypeScript, Tailwind CSS, ESLint
- Installed `@supabase/supabase-js` and `lucide-react`
- Created environment config (`.env.local`, `.env.local.example`)
- Built `lib/supabase.ts` (singleton client with env validation)
- Built `lib/types.ts` (ChatMessage, UserPresence, TypingUser interfaces)
- Built `lib/utils.ts` (generateRoomId ‚Äî 6-char alphanumeric)
- Configured dark theme in `globals.css` using `@theme inline` (void/phantom/ghost color tokens, custom animations, scrollbar-hide utility, reduced-motion media query)
- Set up JetBrains Mono font in `layout.tsx`
- Created room route shell `app/room/[roomId]/page.tsx`

**Key issue:** `create-next-app` rejected uppercase "VoltChat" in package name. Workaround: scaffolded in temp directory and copied files.

### Phase 2 ‚Äî Landing Page & Room Entry
- `components/LandingPage.tsx` ‚Äî Create Room (random ID) and Join Room (manual ID input)
- `components/JoinModal.tsx` ‚Äî Username capture with auto-focus, validation, max 24 chars
- Updated `app/page.tsx` and room page to wire the flow

### Phase 3 ‚Äî Realtime Engine & Core Chat
- `hooks/useChatRoom.ts` ‚Äî Core hook combining Broadcast (messages, nuke) + Presence (online users)
- `components/ChatInterface.tsx` ‚Äî Full chat layout with header, message list, input
- `components/MessageBubble.tsx` ‚Äî Self vs others styling, slide-in animations
- `components/MessageInput.tsx` ‚Äî Text input with send button
- `components/UserSidebar.tsx` ‚Äî Online users list (desktop only initially)
- Auto-scroll logic: scrolls on own message or when near bottom

**Supabase config:** Channel uses `{ broadcast: { self: true }, presence: { key: username } }`. The `self: true` flag means the sender receives their own broadcasts, which simplifies the message flow (single source of truth).

### Phase 4 ‚Äî Ghost Features
- **Typing indicator:** Boolean-only broadcast (not text content). Shows "X is typing..." with bouncing dot animation. Throttled at 100ms, auto-clears after 2s idle. Stale entries swept every 500ms.
- **Visual message decay:** `getMessageOpacity()` ‚Äî newest 5 messages at full opacity, next 10 fade to 0.15, older stays at 0.15. Smooth 2s CSS transition.
- **Nuke animation:** 3-phase sequence ‚Äî shake+flash (0-600ms) ‚Üí message disintegration with staggered delays (600-1400ms) ‚Üí "Room Nuked" overlay (1400ms) ‚Üí redirect to home (2800ms)
- `components/TypingIndicator.tsx`, `components/NukeOverlay.tsx`

**User feedback:** Originally ghost typing showed the actual text being typed. Changed to boolean-only "is typing..." per user preference ‚Äî more privacy-respecting.

### Phase 5 ‚Äî Responsive Polish & Accessibility
- **Mobile sidebar drawer:** `UserSidebar` now has `isOpen`/`onClose` props, mobile backdrop, Escape key close, hamburger menu in header
- **Scroll-to-bottom button:** Appears when scrolled up > 200px
- **Copy room URL:** Button in header with clipboard API + check icon feedback
- **System messages:** Join/leave events rendered as purple italic centered text via `SystemMessage.tsx`
- **ARIA:** `role="log"` on message container, `role="article"` on messages, `role="status"` on typing indicator, `role="alert"` on nuke overlay, `aria-label` on buttons
- **Emoji picker styling:** Custom CSS overrides matching the dark theme (border radius, colors, search input focus ring)

### Post-v1 Updates

#### Room Capacity Limits
- Added optional `maxUsers` field to `UserPresence` and `PublicRoom` types
- `CreateRoomModal` now has a toggle + slider (range 2-50) to set a user limit
- Capacity is passed via `?max=N` URL param and stored in the creator's presence metadata
- All clients read `maxUsers` from the creator's presence (single source of truth)
- When a room is full, joining users see a "Room is full" screen with a back button
- User count and capacity shown in header (`3/10`) and public room cards
- Creator can edit capacity via a settings popover (gear icon in header) with toggle and slider ‚Äî min = current user count

#### Kick & Mute (Creator Moderation)
- **Kick**: Creator can kick any user via the sidebar. Broadcasts a `kick` event; target is disconnected and shown a "You were kicked" screen. Requires confirmation click to prevent accidents.
- **Mute**: Creator can mute/unmute users. Broadcasts a `mute` event; muted users see "You are muted" instead of the message input, and their `sendMessage` is blocked. System messages announce mute/unmute to all users.
- Moderation buttons appear on hover (desktop) or always visible (mobile) in the sidebar next to each non-creator user.

#### Public Room ID Hidden
- Public rooms no longer show the room ID or copy button in the header since they're discoverable through the lobby
